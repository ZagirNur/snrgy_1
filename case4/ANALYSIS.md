# Анализ WEB-систем и архитектуры

Задание: проведите анализ и опишите имеющихся на рынке программного обеспечения информационных систем, построенных по архитектуре WEB-приложений. Оцените и опишите возможности предлагаемых систем по архитектуре WEB-приложений и варианты их использования в компании.

## 1. Что есть на рынке

### 1С-Битрикс24
Российская CRM и корпоративный портал в одном. Работает в браузере, ставить на компьютер ничего не надо, достаточно завести аккаунт. Умеет вести клиентов, сделки, задачи, проекты, внутренний чат, хранилище файлов, календарь сотрудников. Есть бесплатный тариф для маленьких команд и платные - для больших. Плюсы: всё в одном месте, интегрируется с 1С-Бухгалтерией, на русском языке, много готовых шаблонов. Минусы: интерфейс перегружен, при большом числе пользователей начинает тормозить, мобильное приложение не всегда стабильное. Применяют малые и средние компании - продажи, HR, документооборот.

### Odoo
Большая ERP-система с открытым исходным кодом, изначально была бельгийской разработкой. В отличие от Битрикса, Odoo больше про управление ресурсами предприятия - склад, производство, закупки, финансы, точки продаж, HR. Модульная: подключаешь только то, что нужно. Работает тоже через браузер, ставится на свой сервер (community-версия бесплатная) или как облако. Плюсы: открытый код, кастомизируется как хочешь, большое сообщество, сотни готовых модулей. Минусы: для внедрения нужен программист, документация местами запутанная, русская локализация не везде полная. Применяют производственные компании, интернет-магазины, дистрибьюторы.

### amoCRM
Облачная CRM, сфокусированная только на продажах. В отличие от Битрикса здесь нет лишнего, всё заточено под воронку продаж: лиды, сделки, этапы, коммуникации с клиентом через email и мессенджеры. Работает полностью в браузере и мобильном приложении. Плюсы: простой интерфейс, быстро внедряется, хорошо ловит заявки из сайтов и мессенджеров, есть API и интеграции. Минусы: только продажи, для задач и документооборота не подходит, стоимость на пользователя не самая низкая. Применяют отделы продаж и агентства, где главная цель - не упустить лид.

### Metabase
BI-инструмент с открытым кодом для визуализации данных. По факту это веб-приложение, которое подключается к базе данных (Postgres, MySQL, MSSQL, ClickHouse и т.д.) и даёт строить графики и дашборды без знания SQL. Можно задавать вопросы к данным через конструктор или писать SQL напрямую. Плюсы: бесплатно, ставится за 10 минут через Docker, интерфейс понятный, умеет рассылать отчёты по расписанию. Минусы: не заменит полноценный Tableau или Power BI, не умеет сложных расчётов на лету. Применяют там, где нужно быстро показывать менеджерам цифры продаж, остатков, KPI.

## 2. Архитектура WEB-приложений

Все эти системы работают по одной общей схеме - клиент-серверной. Клиент (браузер) не содержит данных и почти не содержит логики, он просто показывает страницы и отправляет запросы. Сервер принимает HTTP-запрос, обращается к базе данных, формирует ответ (HTML, JSON или что-то ещё) и отправляет обратно. База данных живёт отдельно на том же или другом сервере.

В классическом случае у веб-приложения три слоя:
- фронтенд - то, что видит пользователь в браузере (HTML, CSS, JavaScript)
- бэкенд - серверный код, который обрабатывает запросы и работает с БД
- база данных - хранит все данные

Обмен между клиентом и сервером идёт по протоколу HTTP (или HTTPS в защищённом варианте). Каждый URL у приложения соответствует какой-то функции на сервере - это называется маршрутизацией. У нас в приложении, например, URL `/` даёт список туров, `/new` - форму создания, `/{id}/delete` - удаление.

Есть два основных подхода к тому, как формируется страница:
- серверный рендеринг (SSR) - сервер сам собирает готовый HTML и отдаёт его браузеру. Так делает Flask, PHP, старый Битрикс24. Простой подход, хорошо для SEO.
- одностраничники (SPA) - браузер один раз загружает JavaScript-приложение, а дальше подтягивает данные через REST API в формате JSON. Так работают современные интерфейсы типа amoCRM или нового Odoo. Быстро реагирует на действия пользователя, но сложнее разрабатывать.

Главное преимущество веб-архитектуры перед десктопом - ничего не надо устанавливать на компьютер пользователя. Обновление выкатывается один раз на сервер, и все сотрудники сразу работают с новой версией. Доступ есть с любого устройства где есть браузер - компьютер, планшет, телефон. Минус - нужен интернет, и нагрузка от всех пользователей идёт на один сервер.

## 3. Как это использовать в компании

В зависимости от задач компании подходят разные системы:

- **Малый бизнес, отдел продаж** - amoCRM или Битрикс24. Быстро внедряется, не требует своего сервера, помогает не терять заявки и вести клиентов по этапам продажи.
- **Производство, склад, дистрибуция** - Odoo. Здесь важны модули учёта, закупок, производственных цепочек, складских остатков. Под это Odoo заточен.
- **Весь внутренний документооборот и задачи** - Битрикс24. Ставят как единый корпоративный портал, где все задачи, документы, звонки и чаты в одном месте.
- **Аналитика и отчёты** - Metabase. Подключается к базе с рабочими данными и даёт менеджерам дашборды, чтобы не дёргать программистов каждый раз за SQL-запросом.

Часто в одной компании стоит сразу несколько таких систем - например, amoCRM для отдела продаж, Odoo для склада и Metabase для аналитики. Они связываются между собой через API или через общую базу данных.

Наше собственное приложение «Турфирма» - маленький пример того же подхода. Оно решает одну узкую задачу (каталог туров), но построено по тем же правилам: клиент-сервер, HTTP, БД, серверный рендеринг шаблонов. По сути это упрощённый родственник того же Битрикс24, только на 100 строк кода вместо миллиона.
